# Contributing to EtereCitizen

Thank you for your interest in contributing to EtereCitizen. This document provides guidelines for contributing to the project.

## Code of Conduct

By participating in this project, you agree to maintain a respectful and inclusive environment for everyone.

## How to Contribute

### Reporting Issues

- Use [GitHub Issues](https://github.com/icaroholding/EtereCitizen/issues) to report bugs or request features
- Search existing issues before creating a new one
- Include reproduction steps, expected behavior, and actual behavior for bug reports

### Pull Requests

1. Fork the repository
2. Create a feature branch from `main`
3. Make your changes
4. Ensure all tests pass: `pnpm test`
5. Ensure the build succeeds: `pnpm build`
6. Ensure type checking passes: `pnpm typecheck`
7. Submit a pull request

### Development Setup

```bash
git clone https://github.com/icaroholding/EtereCitizen.git
cd EtereCitizen
pnpm install
pnpm build
pnpm test
```

### Project Structure

```
packages/
  common/       — Shared types, schemas, constants
  contracts/    — Solidity smart contracts (Hardhat)
  sdk/          — Core TypeScript SDK
  cli/          — Command-line interface
  mcp-server/   — MCP Server for agent-to-agent tools
  api/          — Hono REST API
  web/          — Next.js web verifier
```

### Build Order

Packages must be built in dependency order:

```
common -> contracts -> sdk -> (cli, mcp-server, api) -> web
```

Running `pnpm build` from the root handles this automatically.

### Testing

- **Vitest** for TypeScript packages: `pnpm test`
- **Hardhat** for smart contracts: `cd packages/contracts && npx hardhat test`
- Write tests for new features and bug fixes
- Maintain or improve test coverage

### Code Style

- TypeScript strict mode
- ESLint + Prettier for formatting
- Run `pnpm lint` before submitting

### Commit Messages

Use clear, descriptive commit messages:

- `add: new feature description`
- `fix: bug description`
- `update: enhancement description`
- `refactor: what was refactored`
- `test: what was tested`
- `docs: what was documented`

## Smart Contract Changes

Changes to Solidity contracts require extra care:

- All changes must include Hardhat tests
- `security.yml` will run Slither on every PR touching `packages/contracts/`; medium+ findings fail the build
- Verify on BaseScan after deployment
- Consider gas optimization
- Review security implications — see [THREAT-MODEL.md](./THREAT-MODEL.md)

## Protocol (Specification) Changes

[`SPEC.md`](./SPEC.md) is normative. Changes to protocol semantics MUST go through:

1. **Open a spec issue first** using the "Specification issue" template. Describe the ambiguity or new capability and why it matters.
2. **Discuss on the issue.** Non-trivial spec changes need review before code lands.
3. **Write the spec change in the PR alongside any implementation changes.** Do not merge implementation ahead of spec; the reference SDK should never diverge from `SPEC.md`.
4. **Update conformance artifacts:**
   - If the change affects a wire format, update or add a JSON Schema in `spec/schemas/`.
   - If the change affects a deterministic algorithm, add a test vector in `spec/test-vectors/` — generated by running the reference implementation, NOT by hand.
   - Run `pnpm --filter @eterecitizen/conformance test` and confirm all vectors pass.
5. **Architecture Decision Record.** For non-obvious design choices, add an ADR in `docs/adr/` following the template of the existing entries.
6. **Version.** Any observable wire-format or contract-interface change requires a minor or major version bump per SPEC §19. Note the version in your PR description.

## Adding a New Implementation (e.g., Python/Go SDK)

We welcome alternative-language implementations. To claim conformance:

1. Implement the protocol following `SPEC.md`.
2. Wire your implementation to `@eterecitizen/conformance` by writing a thin adapter that exposes the functions listed in the package README.
3. Publish your adapter's conformance report alongside your release. 24/24 passing is the baseline.
4. Open a PR to `README.md` listing your implementation under "Alternative implementations" once it passes the conformance suite.
